# 蒸馏质量评估：三个关键问题

> 基于 Shadow-Folk 实际记忆数据（shadowfolk_summaries_week.csv + shadowfolk_observations_week.csv），验证蒸馏管线是否能回答三个核心问题。

---

## 一、什么叫好结果？

从记忆数据中提取出的"好结果"定义，不是主观判断，而是有具体可验证标准的：

### MCP 端点部署
**好结果标准**：三个验证场景全部通过
1. `initialize` 握手成功，返回 `protocolVersion: "2025-03-26"`, `serverInfo: shadowfolk-http-mcp`
2. 无效 Token 被拒，返回 `code: -32000, "Unauthorized: invalid or missing API token"`
3. `tools/list` 正确返回 13 个工具（get_shadow, search_memory, ask_shadow 等）

> 数据来源：observations #11700, #11699

### 他人接入
**好结果标准**：零本地依赖
- 只需粘贴一段 JSON 配置到 `~/.cursor/mcp.json`
- 不需要 clone 仓库、不需要安装 Node.js、不需要 `npm install`
- 分发成本从"搭环境"降低为"粘贴配置"

> 数据来源：summaries #1375, observations #11938

### 架构迁移（stdio → HTTP）
**好结果标准**：最小改动
- stdio 入口保留不删（降低回退风险）
- HTTP 新增不破坏旧功能
- 不改 Nginx 配置
- 不改数据库
- 不改 Docker Compose

> 数据来源：observations #11943, #11702

### 鉴权排查
**好结果标准**：用服务端事实做判定，不信客户端表象
- 服务端日志无异常 + 所有请求返回 200 → 服务正常
- 实际工具调用截图可用 → MCP 功能正常
- Cursor 红色报错是 SSE 轮询噪音 → 与真实故障无关

> 数据来源：observations #12252

## 二、常见错误有哪些？

从记忆数据中反复出现的错误模式，按频率和影响排序：

### 高频错误

| 错误 | 出现次数 | 具体表现 | 影响 |
|------|---------|---------|------|
| **Bearer 前缀缺失** | ≥2 个用户 | Authorization 只传了 token 值，没加 `Bearer ` 前缀 | 永久 401，看起来像服务挂了 |
| **CRLF 行尾** | 连续 3 条记录 | Windows 编辑的脚本在 Linux 报 `$'\r': command not found` | 脚本完全无法执行 |
| **多层引号嵌套** | 3 次失败 | PowerShell → SSH → Bash → Python 四层转义崩溃 | 每一层都会吃掉或转义引号，手写不可能正确 |

### 中频错误

| 错误 | 出现次数 | 具体表现 | 影响 |
|------|---------|---------|------|
| **lockfile 不同步** | 1 次部署失败 | 改了 `package.json` 没跑 `pnpm install`，`--frozen-lockfile` 构建报错 | 阻断整个部署流程 |
| **混淆认证层级** | 贯穿 6 条记录 | SSH 层 `authz success` 与应用层 401 同时出现，看起来矛盾 | 误判问题方向，浪费排查时间 |

### 低频但隐蔽的错误

| 错误 | 出现次数 | 具体表现 | 影响 |
|------|---------|---------|------|
| **用错查询工具** | 1 次 | 用 `search_knowledge` 查 PAK 返回空，应该用 `search_memory_like` | 误判为 MCP 故障 |
| **被客户端噪音误导** | 1 次 | Cursor 红色报错是 SSE 轮询探测，不是真正故障 | 如果被带偏可能浪费数小时 |

## 三、哪些判断节点最关键？

从记忆数据中提取出 6 个改变了事情走向的关键判断。每一个选错，后果都很具体：

### 判断 1：无状态 JSON vs 有状态 SSE

| | 选了（正确） | 没选 |
|---|---|---|
| **方案** | 无状态 JSON 模式 | 有状态 SSE 会话 |
| **代价** | 无 | 需维护 session Map + 清理逻辑，需改 Nginx proxy_buffering 和超时 |
| **为什么正确** | 现有 13 个工具全部是查询返回型，没有流式输出或主动推送需求 |

> 数据来源：observations #11649, #11648

### 判断 2：路由注册在 TOF 之前还是之后

| | 选了（正确） | 没选 |
|---|---|---|
| **方案** | `/api/mcp` 挂载在 TOF 认证中间件**之前** | 挂在 TOF 之后 |
| **代价** | 无 | 所有 MCP 请求被 TOF 拦截，返回"请先登录 TOF 认证中心"，Bearer Token 永远不生效 |
| **为什么正确** | MCP 端点走 Bearer Token 认证，和 TOF 是两条独立链路，不应被 TOF 中间件干扰 |

> 数据来源：observations #11683, #11687

### 判断 3：身份从 Token 取还是从工具参数取

| | 选了（正确） | 没选 |
|---|---|---|
| **方案** | 只从 Bearer Token 解析 `personId` | 信任工具参数中传入的 person_id |
| **代价** | 无 | 任何人可以在工具参数中伪造 person_id，造成越权访问 |
| **为什么正确** | 工具参数中的 person_id 仅用于指定"查询目标对象"，不是调用者身份 |

> 数据来源：observations #11651, #11650

### 判断 4：抽离共享模块还是复制代码

| | 选了（正确） | 没选 |
|---|---|---|
| **方案** | 新建 `tools.ts` + `handlers.ts`，stdio 和 HTTP 共用 | 给 HTTP 入口复制一份工具定义 |
| **代价** | 多花 30 分钟做拆分 | 两套代码逐渐分叉，后续每次改工具要改两个地方 |
| **为什么正确** | 工具定义已有 13 个，且持续增长，分叉的维护成本会指数增长 |

> 数据来源：observations #11702, #11651

### 判断 5：Cursor 红色报错是不是真故障

| | 选了（正确） | 没选 |
|---|---|---|
| **判断** | SSE 轮询探测噪音，不影响 POST 工具调用 | 花时间排查这个"错误" |
| **代价** | 无 | 可能浪费数小时排查一个不存在的问题 |
| **怎么判断的** | 先看服务端日志（全部 200），再看实际工具调用截图（成功），反推客户端表现是噪音 |

> 数据来源：observations #12252

### 判断 6：search_knowledge 空结果是 bug 还是数据问题

| | 选了（正确） | 没选 |
|---|---|---|
| **判断** | 知识库中没有 PAK 内容，不是 MCP 故障 → 换 `search_memory_like` | 继续排查 MCP 服务为什么返回空 |
| **代价** | 无 | 在正常工作的服务上反复排障，方向完全错误 |
| **怎么判断的** | `search_knowledge` 调用本身返回成功（200），只是结果集为空 → 问题在数据不在服务 |

> 数据来源：observations #12252

---

## 总结：蒸馏管线能自动回答这三个问题吗？

| 问题 | 能否从原始数据提取 | 难点 |
|------|------------------|------|
| 什么叫好结果 | ✅ 能 | 需要从 summaries 的 `completed` 和 `learned` 字段中提取验证标准 |
| 常见错误有哪些 | ✅ 能 | 散布在 20+ 条 observations 中，需要按错误类型聚合并计数 |
| 哪些判断节点最关键 | ✅ 能，但最难 | 需要**跨记录关联推理**——从设计讨论、部署失败、最终方案三类记录中交叉提取"选了 A 而不是 B"的决策点 |

这正是蒸馏管线 Layer 2（叙事重建）要解决的核心问题：不只是压缩事实，而是**重建决策上下文**。
