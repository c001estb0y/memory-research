# 公司大脑（三）：交互记忆

> 原文：[Company Brain, Part 3: Interaction Memory](https://nanothoughts.substack.com/p/company-brain-part-3-interaction)
> 作者：Ashwin Gopinath · 2026年5月6日

---

几乎所有公司里重要的事情都发生在会议、消息或邮件中。

这听起来夸张，直到你开始列举决策实际被做出的地方。一个客户在电话里投诉。一个销售代表承诺了一个变通方案。一个经理说项目被阻塞了，但只是在有人问了两次之后才说的。一个创始人在五分钟的 Slack 交流中改变了优先级。法务附带条件地批准了某件事。工程团队同意发布，但前提是一个假设成立。

后来，一些制品出现了。一个工单。一条 CRM 更新。一个路线图笔记。发布文档里的一行字。到那时，真正的上下文已经被压缩了。

上一篇文章讲的是[事实记忆：发生了什么](https://x.com/ashwingop/status/2049885545288077720)，来源在哪里，谁负责，什么时候改变了，东西之间如何关联。交互记忆不同。它不是制品的记忆。它是在制品存在之前，人与人之间发生了什么的记忆。

它的目标是保留事实记忆通常丢失的部分：**为什么**某件事发生了，人们期望工作如何完成，什么约束条件重要，哪个假设是脆弱的，以及什么没有被说出来。交互是组织的思维链。它是一个群体如何从部分信息走向判断的轨迹。

这个区别很重要，因为公司不在数据库里做大多数决策。它们在对话中做出决策。数据库记录的是事后的结果。一个 CRM 字段可能说一个账户有风险，但客户电话解释了真正的异议。一份路线图文档可能说发布推迟了两周，但会议解释了权衡。一个工单可能说需要 SSO，但销售对话解释了为什么它重要以及谁做出了承诺。

人与人之间的交互仍然是人与 Agent 交互的上游。Agent 很重要，Agent 轨迹会随着时间变得更重要。但今天，大多数 Agent 工作继承了来自人类对话的目的。Agent 可能起草邮件、更新工单、总结对话或执行工作流。工作存在的原因通常来自人们彼此之间的交谈。

如果事实记忆是公司记住自己的制品，交互记忆就是公司记住意义是如何被创造的。

一份转录不够。一份摘要不够。即使是一份好的会议记录也只是交互的表面表示。困难的问题是解读。实际上决定了什么？什么被暗示了但没有说？人们在什么问题上有分歧？哪个异议是真实的？哪个承诺是随口做出的但以后会变得重要？

**本体论**是这个问题的名称。本体论是一个系统用来理解一个领域的概念和关系的集合。在公司中，本体论决定了对话中的某个东西是决策、承诺、异议、升级、依赖、假设、客户痛点、负责人、先例还是未决问题。这些标签不仅仅是元数据。它们决定了什么被记住。

拿一个会议中的普通句子来说："如果法务签字并且 Acme 对 beta 限制没意见，我们可以这周五发布。"

一份转录存储了这个句子。一份会议摘要可能说团队讨论了周五发布。交互记忆必须看到下面的结构。从产品视角，这是一个带约束条件的发布计划。从法务视角，这是一个审批依赖。从客户视角，这是对 Acme 的一个有条件的承诺。从销售视角，这可能是一个交易风险。从行动视角，它包含后续行动。从高管视角，这实际上不是一个已关闭的决定。

人类自然地做到这一点。根据客户、项目、说话者、账户的历史，以及如果解读错误会发生什么，我们对同一句话有不同的理解。现有系统通常做不到。它们存储句子一次，附上一个摘要，然后寄希望于检索能在之后恢复意义。

交互记忆不能是一个静态的笔记档案。交互的意义取决于用来解读它的本体论。改变本体论，同一个交互可以意味着不同的东西。客户的一个随意异议可以后来成为流失风险的证据。架构会议上的一个技术担忧可以后来解释为什么发布延迟了。邮件中的一行审批可以后来成为先例。

公司必须能够**重读自己的过去**。

人类一直在这样做。某个看起来只是小小抱怨的东西，在又有三个客户重复之后变得重要了。一个模糊的担忧变成了所有人都应该听进去的东西。一个看起来已经定下来的决定，在缺失的假设浮现之后变得明显是有条件的。一个真正的公司大脑应该能够做到某种版本的这件事。

这里**上下文图谱**很重要。上下文图谱是连接人员、团队、客户、项目、承诺、决策、风险、假设、依赖和时间的结构。交互记忆应该更新这个图谱。一次会议不应该只产生一个笔记。它应该改变公司对自己相信什么、承诺了什么、什么仍未解决、以及接下来应该发生什么的认知地图。

画成图的版本是干净的。真实的版本是混乱的。人们模棱两可。他们礼貌地表达不同意见。他们暗示所有权而不正式分配它。他们做决定却不宣布已经做了决定。他们用"跟上次一样的问题"或"那个企业客户的事"来引用之前的上下文。他们在同一个十分钟内在战略、情绪、执行和政治之间切换。

一个有用的系统必须知道什么该记住，什么该忽略。记忆太少，公司不断忘记事情为什么发生。记忆太多，系统变得嘈杂或令人毛骨悚然。交互记忆非常接近人们如何思考、谈判、争论和改变主意。如果产品做错了，感觉像是监控。如果做对了，感觉像是公司终于不再丢失线索了。

这意味着权限和边界不是细节。有些交互是私人的。有些是团队上下文。有些是公司记录。有些可以被总结但不能被引用。有些应该贡献给聚合记忆而不需要向更广泛的人群暴露原始对话。公司大脑必须理解这些区别，否则人们不会信任它承载最重要的上下文。

这也意味着界面不能只是一个会议录音器。录音、转录和行动项是有用的，但它们不是这个品类。界面应该在交互之前、期间和之后提供帮助。在客户电话之前，它应该浮现之前的承诺、未解决的问题、决策历史和未回答的问题。在会议之后，它应该识别决策、假设、风险和后续行动。几周后，它应该注意到跨交互的模式，这些模式没有任何一个人能在脑子里记住。

对于一线贡献者，交互记忆可能回答："上次架构讨论我们决定了什么，哪些假设仍然需要验证？"答案不应该是一份转录。它应该说什么改变了，什么仍然悬而未决，谁负责下一步，以及哪些之前的讨论是相关的。

对于经理，问题可能是："我的团队本周在客户电话和内部会议中做出了哪些承诺？"答案应该将人类的承诺连接到工单、负责人、截止日期和风险。它还应该注意到尴尬的情况：被暗示但从未分配的承诺，或者不同人理解不同的决策。

对于 CEO，问题往往更宽泛："哪些团队在基于不同的假设做决策？"这就是交互记忆变得远不止于记笔记的地方。它可以显示销售认为某个功能已承诺，产品认为它是探索性的，工程认为它被阻塞了，而客服认为它已经迟了。没有任何仪表盘能清晰地展示这一点。真相存在于交互中。

主动版本是有趣的那个。公司大脑应该注意到同一个异议出现在三次客户电话中。它应该注意到两个团队对同一个指标使用了相互矛盾的定义。它应该注意到一个决策不断被重新讨论，因为最初的权衡从未被清晰记录。它应该注意到一个负责人被暗示了但从未被指定。它应该注意到一个升级已经悄悄变成了一个产品信号。

这是记忆和行动之间的桥梁。事实记忆可以告诉你什么存在。交互记忆告诉你人们的意图、辩论的内容、承诺的事项和未解决的问题。它保留了很少进入制品的组织推理。没有它，Agent 将继续在不完整的上下文下游运作。它们可能检索工单、起草邮件或更新 CRM，但它们不会知道这项工作重要的人类原因。

这就是为什么我认为公司大脑问题从人类沟通开始。不是因为文档和数据不重要。它们显然很重要。但公司的意义是在交互中被创造的，然后才在其他任何地方被正式化。

交互记忆是公司大脑的第二层。它将会议、消息、邮件、电话、投诉和辩论转化为结构化记忆。但它仍然没有闭合循环。一旦公司记住了发生了什么并理解了交互的意义，它就必须协调接下来发生什么。

那就是第三层：行动记忆。
