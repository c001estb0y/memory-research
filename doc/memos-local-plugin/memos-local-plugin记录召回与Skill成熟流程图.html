MemOS Local Plugin / Standalone Flowchart

记录、召回与 Skill 成熟

这个文件单独解释 memos-local-plugin 的完整运转：执行数据如何被记录成 L1 trace， 如何通过评分标记“有用”，如何在 turn start 被召回，最后如何把共性经验升级成成熟 Skill。

总览 记录链路 评分机制 Policy 成熟 Skill 成熟 召回链路 人打分还是自动打分
FLOW 01

一张总图：从执行记录到未来召回

Agent 执行

OpenClaw / Hermes 产生用户输入、assistant 输出、thinking、工具调用和工具结果。

→
Adapter 标准化

OpenClaw 走进程内 TS hooks；Hermes 走 Python MemoryProvider + JSON-RPC bridge。最终都转成统一 DTO。

L1 Trace

记录每步事实：state、action、tool outcome、reflection、summary。

→
评分回传

先评 reflection 质量 alpha，再评任务结果 R_human，最后回传每步 value 与 priority。

L2 / L3 提炼

高价值重复 trace 归纳为 L2 policy；多个 policy 抽象出 L3 world model。

→
Skill 结晶

高 support、高 gain、可验证的 active policy 被包装为 candidate Skill，再通过试用和反馈晋升。

下一轮 Turn Start

当前任务开始前，core 先检索记忆。

→
三层召回注入

Tier 1 Skill、Tier 2 Trace/Episode、Tier 3 World Model 被排序、过滤、渲染成 InjectionPacket 注入 prompt。

FLOW 02

记录链路：插件如何拿到执行数据

OpenClaw

before_tool_call / after_tool_call

记录工具名称、输入、输出、错误码、耗时和成功状态。

agent_end

Hermes

实现 Hermes 的记忆接口，作为 Python 侧适配层。

post_tool_call / post_llm_call

捕获工具执行结果和模型输出，累积成当前 turn 的证据。

JSON-RPC bridge

通过 bridge.cts 把 Python 事件发给 TypeScript MemoryCore。

进入 core 后统一成 L1 证据

userText

用户请求、环境观察、上一轮上下文。

agentText

Agent 当步输出或最终回答。

agentThinking

FLOW 03

评分机制：哪些数据被标记为“有用”

第一层：每步 reflection 的质量分 alpha

Capture 的 reflect pass 会让 LLM 看完整 step 或完整 episode，对每步 reflection 打分。 alpha 衡量这条反思是否忠实、具体、有因果解释、能迁移。

  • alpha=0：空洞、不可信、不可用。
  • alpha 越高：后续回传越相信这一步的因果解释。
  • 这不是用户满意度，而是 step 级证据质量。

第二层：整个任务的 R_human

Reward 模块用 task summary 加用户反馈打任务分。显式反馈越清楚，信号越强； 没有显式反馈时，LLM rubric 会保守估计目标完成度、过程质量和用户满意度。

R_human = 0.45 * goal_achievement + 0.30 * process_quality + 0.25 * user_satisfaction

第三层：把任务分回传到每条 trace

V_T = R_human V_t = alpha_t * R_human + (1 - alpha_t) * gamma * V_{t+1} priority = max(V_t, 0) * time_decay

这一步之后，每条 L1 trace 都不再只是历史记录，而有了 value 和 priority。高 priority 的正向 trace 更容易被召回；负价值 trace 不会被删除， 但更适合作为失败反例和 decision repair 输入。

FLOW 04

从高价值 trace 到成熟 L2 Policy

相似 trace 聚类

按 state/action、embedding、标签和工具等信号，把相似执行片段放入 candidate pool。

→
L2 induction

LLM 用 l2.induction prompt 归纳“遇到 X，采用 Y，并注意 Z”的过程策略。

计算 support

统计有多少 episode / trace 支持这个策略。样本不足时继续保持 candidate。

→
计算 gain

比较使用该策略的路径和未使用策略的基线，判断它是否真的带来收益。

状态迁移

support >= minSupport 且 gain >= minGain 时，policy 从 candidate 变 active。

→
归档条件

如果 active policy 后续 gain < archiveGain 或 support 归零，就归档，不再作为正向策略使用。

指标 它回答的问题 怎么来
support 这个策略是不是有足够证据支撑？ 从源 episode / trace 计数而来。
gain 这个策略是不是比普通路径更有收益？ mean(V_with_policy) - blended_mean(V_without_policy)。
status 这个策略当前能不能参与后续 Skill 结晶？ 由 support、gain、archiveGain 等门槛决定。
FLOW 05

从 active Policy 到成熟 Skill

Eligibility

policy 必须是 active，且 gain、support 达标，并有正向 success anchor。

Crystallize

LLM 用 skill.crystallize prompt 把 policy、evidence、counter examples 写成 Skill draft。

Verify

确定性校验工具覆盖率和证据共振，防止 LLM 编造工具或写出脱离证据的 Skill。

Candidate Skill

通过校验后仍先是 candidate，不会立刻作为成熟能力完全启用。

Skill 的可靠度 eta

Skill 的成熟度不是只看 LLM 写得好不好，而是看后续试用表现和用户反馈。 eta 是技能可靠度，受 policy gain、trial pass/fail、用户正负反馈共同影响。

候选到成熟的门槛

当 candidate Skill 的 trialsAttempted >= candidateTrials，并且 eta >= minEtaForRetrieval，才会晋升为 active。如果 eta < archiveEta，会被归档。

FLOW 06

召回链路：有用经验如何回到下一次 prompt

Turn Start

用户发起新任务，adapter 在 prompt build 前调用 core.onTurnStart()。

→
构造检索查询

结合当前 user ask、workspace、namespace、工具上下文和历史状态。

Tier 1：Skill

优先找 active Skill。命中时直接给使用时机、步骤、参数、preference 和 anti-pattern。

→
Tier 2：Trace / Episode

按相似度、关键词、priority 和时间衰减召回具体历史步骤，补细节和证据。

Tier 3：World Model

召回项目结构、环境约束、领域规律，不直接给动作，但减少重复探索。

→
InjectionPacket

把少量高价值记忆渲染成 prompt prelude，交给 Agent 作为当前任务上下文。

ANSWER

到底是 human 打分，还是 agent 自动打分？

准确答案是：混合评分。用户反馈是强信号，但系统不会要求用户每轮都手动打分。 没有用户反馈时，插件会用 LLM rubric、启发式规则和公式回传自动估计有用性。

不是“LLM 觉得有用就变 Skill”，而是 evidence → score → policy gain → verifier → trial → eta 的多级门槛。
信号 来源 人工还是自动 影响范围
alpha LLM reflection judge 自动 决定 step 反思是否可信，影响价值回传。
R_human 用户反馈 + LLM rubric + heuristic fallback 混合 决定整个 episode 的任务成败信号。
value / priority backprop 公式 自动 决定 trace 是否容易被未来召回。
support / gain 证据计数 + 对照价值计算 自动 决定 L2 policy 是否从 candidate 变 active。
eta policy gain、trial 结果、用户正负反馈 混合 决定 Skill 是否晋升 active 或被 archived。
