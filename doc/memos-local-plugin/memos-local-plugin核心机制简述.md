# MemOS Local Plugin 核心机制简述

`memos-local-plugin` 的核心不是简单记录每项任务的内容，而是把每次 Agent 执行变成可评分、可归因、可复用的经验资产。

## 1. 先记录发生了什么

系统首先记录任务执行事实：

```text
用户说了什么
Agent 回答了什么
调用了哪些工具
工具结果是什么
任务最后如何收尾
```

对应的数据主要是：

| 数据 | 含义 |
| --- | --- |
| `trace` | 单步或单轮执行痕迹 |
| `episode` | 多条 trace 组成的一次任务单元 |

这一步类似日志，但比普通日志结构化。

## 2. 再判断做得怎么样

系统不会平等看待所有记录，而是会给任务和步骤打分：

| 信号 | 含义 |
| --- | --- |
| `feedback` | 用户显式或隐式反馈 |
| `r_task` / `R_human` | episode 级任务总评分 |
| `alpha` | trace 反思质量权重 |
| `value` | trace 对任务成败的贡献 |
| `priority` | trace 未来是否值得召回 |

直觉上：

```text
整个任务是否满足用户目标？
哪一步真正有贡献？
哪一步导致失败？
哪条记录以后还值得拿出来参考？
```

这一步的重点是归因，而不是只保存结果。

## 3. 最后总结可复用模式

当系统积累了足够多的 trace、episode 和反馈后，会进一步提炼可复用经验：

| 层级 | 含义 |
| --- | --- |
| `policy` | 遇到类似情况，下次应该怎么做 |
| `world_model` | 当前环境、工具、约束和用户偏好的稳定认知 |
| `skill` | 从成熟 policy 打包出的可调用流程 |

也就是说，系统最终想学到的是：

```text
哪些行为有效？
哪些行为不可靠？
哪些模式值得下次复用？
哪些流程可以沉淀成技能？
```

## 4. 和普通日志的区别

普通日志更像：

```text
这次任务发生了 A、B、C。
```

MemOS 的目标更像：

```text
这次任务发生了 A、B、C；
用户是否满意？
哪一步贡献最大？
哪一步应该避免？
这种模式以后能否复用？
能不能变成 policy 或 skill？
```

## 5. 一句话总结

`memos-local-plugin` 的核心不是“记住任务内容”，而是从任务执行中学习：

```text
记录事实 -> 评分归因 -> 提炼模式 -> 复用经验
```

它要沉淀的不是流水账，而是可在下一次任务中真正改变 Agent 行为的经验。
