Part 01 — 元问题

为什么需要记忆？

所有项目的出发点都是同一个根本矛盾

AI Agent 的上下文窗口是有限且易逝的，但人类期望它拥有跨会话、跨时间的持久认知能力。

"更长的上下文窗口"不是答案 — 即便窗口无限大，没有组织、没有遗忘、没有提炼的信息堆积，只是噪声的放大器，不是记忆。

五个第一性问题

📝
编码问题
原始交互如何变成结构化记忆？
感知 → 工作记忆
🕸️
组织问题
记忆单元之间如何建立关系？
语义网络 / 图式
🔍
检索问题
如何在正确的时刻召回正确的记忆？
线索激活 / 联想
🔄
演化问题
记忆如何随时间更新、合并、遗忘？
巩固 / 干扰 / 遗忘
⚖️
经济问题
如何在有限 token 预算下传递最大信息量？
注意力分配 / 工作记忆容量
Part 02 — 五层模型

成长型系统的认知成熟度

一个精确的评判框架，用于衡量记忆系统的"认知成熟度"

5 迁移与行为改变
未来任务里，它真的开始表现出行为改变与知识迁移
覆盖率: 几乎空白 — 仅 MemOS 预留占位
4 治理
哪些该保留？哪些该合并？哪些彼此冲突？哪些已过时？
覆盖率: 少数项目探索 — graphiti / openclaw / supermemory
3 提炼
从原始经历里抽出更高层的经验、规律、偏好、失败模式 — 走到这一步，才接近"学习"
覆盖率: 部分项目触及 — m_flow / cognee / ClawXMemory / OpenViking
2 检索
下一次能想起来。这很有用，但本质是 recall，不是 growth
覆盖率: 所有项目 — 投入工程量最多的层级
1 记录
把事情留下来。这是地基，没有这层什么都别谈
覆盖率: 100% — 三条路线: LLM抽取 / 规则捕获 / 文件即记忆
核心洞察：大多数"记忆系统"止步于第 2 层（检索），少数触及第 3 层（提炼），极少数开始探索第 4 层（治理），而第 5 层（迁移）目前几乎是空白。

Part 03 — 检索范式

五种检索范式

从纯向量近邻到 LLM 推理检索，认知深度逐级递增

1
纯向量近邻
mem0 · supermemory
embedding → ANN → top-k，最简单直接的相似性检索
认知对应：相似性联想（最浅层）
2
混合检索
cognee · gbrain · mempalace · claude-mem
向量 + FTS/BM25 + metadata 过滤，多信号融合
认知对应：多线索并行激活
3
图路径传播
m_flow · graphiti
向量锚点 → 图遍历 → 路径成本评分，沿有意义的关系扩展
认知对应：深层语义联想（最接近人类）
4
LLM 推理检索
ClawXMemory · memU
多跳 LLM + 充分性门控，模型逐跳推理判断
认知对应：推理性回忆（最慢但最准）
5
级联渐进
OpenViking · 蒸馏系统
L0→L1→L2 逐层深入，先概要后细节
认知对应：自然叙事模式
所有检索都可抽象为三阶段： 候选生成（Recall） → 精排（Rank） → 预算裁剪（Budget）

核心反思：Similarity ≠ Relevance

向量空间只编码了语义表面（"说了什么"），没有编码认知结构（"为什么重要、与什么相关、在什么条件下成立"）

Similarity（相似性）

对称的：sim(A,B) = sim(B,A)
无语境的：与查询目的无关
只捕获"像"的关系
传递性不可靠
回答：这两段话说的是不是同一件事？

Relevance（相关性）

不对称的：rel(A→B) ≠ rel(B→A)
依赖语境：完全取决于"为什么问"
捕获因果、时序、对比、条件等
可沿有意义的路径传导
回答：在当前语境下，是否应该被想起来？

四个典型失败场景

1
因果关系 — "像"的不是"该想起来的"
"为什么部署后 API 延迟从 50ms 飙到 2000ms？"
❌ 向量近邻

"API 延迟监控配置方法" — 语义最像但没回答"为什么"

✅ 图路径传播

"误删 Redis 连接池" → "重连风暴致 P99 飙升" — 因果链直接相关

2
对比关系 — 最相关的恰恰是"最不像"的
"React 和 Vue 在状态管理上有什么区别？"
❌ 向量近邻

只找到 React 状态管理文档 — 单边信息

✅ 图路径传播

useState → "对比方案"边 → Vue reactive() — 跨域连接

3
时序关系 — "之前"和"之后"向量几乎相同
"我们现在用的什么数据库版本？"
记忆中：3月"PostgreSQL 14" / 4月"已迁移到 PostgreSQL 16"
❌ 向量近邻

两条相似度几乎相同 — 无法区分哪条当前有效

✅ 时序图 (graphiti)

旧边 invalid_at=4月15日 — 时序感知，只返回有效边

4
情景关联 — 一个联想唤起整段记忆
"CORS 报错，但配置看起来没问题"
记忆中：一个月前处理过"看似 CORS 实际是证书过期"的排查
❌ 向量近邻

"CORS 配置教程" — 通用信息

✅ m_flow 锥形图

CORS → "误导性错误表现" → 完整排查 Episode — 联想链唤起

Part 04 — 设计原理

六大底层共性原理

从第一性原理出发，提炼 15 个项目的底层设计共性

双轨存储
Dual-Track Storage
几乎所有项目都采用"向量 + 结构化"的双轨存储。向量捕获语义相似性（"像什么"），结构化捕获关系与约束（"是什么、与什么相关、何时有效"）。两者不可互相替代 — 与人类大脑中海马体（情景记忆）和新皮层（语义网络）的双系统同构。
LLM 嵌入循环
LLM-in-the-Loop
LLM 不仅用于最终回答，还深度嵌入记忆的每个阶段：写入时（事实抽取）、检索时（策略选择、充分性判断）、演化时（冲突决策、知识整理）。LLM 充当"语义胶水"，在每个需要理解自然语言语义的环节提供判断能力。
渐进式披露
Progressive Disclosure
所有成熟系统都实现了"先概要、后细节"。直接对应人类认知中的"注意力漏斗"。在 token 经济学下，渐进式披露是最优的信息传递策略：以最少的 token 传递最高的信息密度。
L0 (~100 token) → L1 (~1-2k token) → L2 (原文) 概要/精华 结构化经验 完整证据
溯源不可断
Provenance Chain
成熟的系统都维护从最终记忆到原始来源的可追溯链。没有溯源的知识是不可信的 — 与科学研究中"可重复性"、法律中"证据链"是同一原则：任何结论都必须能追溯到原始证据。
事件驱动摄入
Event-Driven Ingestion
IDE 类记忆系统都采用 Hooks / 事件驱动的非侵入式数据采集。记忆的采集不应中断工作流 — 与操作系统中"中断驱动 I/O"的设计哲学一致：后台异步捕获，主线程不阻塞。
统一生命周期
Unified Lifecycle
所有系统的记忆都遵循同一个状态机，与心理学中的 Atkinson-Shiffrin 记忆模型（感觉记忆 → 短时记忆 → 长时记忆）和遗忘曲线高度同构。

Part 05 — 组织拓扑

记忆的组织拓扑 — 五种范式

15 个项目展现出五种不同的记忆组织方式

扁平向量空间
mem0 · supermemory
所有记忆嵌入同一空间，靠 metadata 过滤。简单快速，但丢失关系。
分层金字塔
ClawXMemory · OpenViking · memU
按信息密度/抽象级别分层，从顶层摘要到底层原文。模仿"概要→细节"认知模式。
知识图谱
graphiti · cognee · mempalace
实体 + 关系 + 时序的图结构。共同抽象：Entity → Predicate → Entity + 时间戳。
认知锥形
m_flow
四层锥形结构：Episode → Facet → FacetPoint → Entity。范式 2 与范式 3 的统一。
空间隐喻
mempalace · MemOS · OpenViking · gbrain
用物理/虚拟空间概念组织：Wing/Room/Drawer、MemCube、viking://。本质是命名空间 + 层级 + 横向连接。
Part 06 — 行业洞察

关键洞察与趋势

三代演化

第一代 — 向量存取
向量存取 + LLM 抽取
第 1-2 层（记录+检索）
mem0 supermemory
第二代 — 结构化增强
向量 + 图/结构化 + 时序
第 1-3 层（+提炼），触及第 4 层（治理）
graphiti cognee mempalace
第三代 — 认知引擎
图作为评分引擎 + 认知锥形
第 1-3 层深度突破，第 4 层初步
m_flow
洞察 01
"联想 ≠ 搜索" 是最深刻的范式反思
搜索给你"最像的片段"，联想给你"最该被想起来的那个情景"。图路径传播是目前最接近"联想"的工程实现，但计算成本和复杂度远高于纯向量方案。
洞察 02
第 5 层是整个赛道的"无人区"
当前所有系统都是"外挂式记忆" — 通过 prompt injection 注入。注入 ≠ 内化：每次都需显式检索+注入，模型本身没有"学会"。MemOS 是唯一在架构上为第 5 层预留了位置的项目（LoRA 参数记忆），但目前为纯占位符。
洞察 03
IDE 记忆是最活跃的应用前线
数据源天然结构化（工具调用、文件编辑）、反馈信号明确（编译/测试通过/失败）、用户感知直接（开发者立即感受到"Agent 是否记住了"）。claude-mem、codebuddy-mem、ClawXMemory 等构成最密集的创新区。
